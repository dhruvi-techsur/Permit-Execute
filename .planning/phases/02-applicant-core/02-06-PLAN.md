**Form schema** (zod):
```typescript
const permitFormSchema = z.object({
  permitType: z.enum(['construction', 'zoning_variance', 'event_permit', 'demolition', 'renovation', 'signage'], {
    required_error: 'Permit type is required',
  }),
  title: z.string().min(3, 'Title must be at least 3 characters').max(200, 'Title must be 200 characters or less'),
  description: z.string().min(10, 'Description must be at least 10 characters').max(5000, 'Description must be 5000 characters or less'),
  siteAddress: z.string().min(5, 'Site address is required'),
  contactPhone: z.string().optional(),
});
type PermitFormValues = z.infer<typeof permitFormSchema>;
```

**Page logic**:

1. Detect mode:
   - `const { id } = useParams()` — if `id` is undefined → create mode (new application)
   - Edit mode → on mount: `getPermit(id)` to pre-fill form; show skeleton during load

2. Create mode flow:
   - On first form interaction (or on mount): call `createPermit({ permitType: 'construction', title: '', ... defaults })` to get a permit ID → store in local state as `permitId`
   - Actually: to avoid creating empty permits, only call `createPermit` when the user first clicks "Save Draft" or on first auto-save trigger
   - Alternative approach (simpler, use this): In create mode, form data is held in local state. When user clicks "Save Draft" for the first time, call `createPermit(formValues)` → get `permitId` → navigate to `/applications/${permitId}/edit` (which pre-fills the form). Subsequent saves call `updatePermit(permitId, changes)`.

3. Auto-save (edit mode only):
   - `useEffect(() => { const timer = setTimeout(() => { if (isDirty && permitId) { saveDraft(); } }, 2000); return () => clearTimeout(timer); }, [watchedValues])` — 2000ms debounce
   - `saveDraft()` calls `updatePermit(permitId, formValues)` and sets `savedAt = new Date()` state
   - Display: subtle gray text "Saved just now" / "Saved 2 min ago" near Save Draft button — using `formatDistanceToNow(savedAt, { addSuffix: true })` from date-fns — NOT a toast

4. Unsaved changes guard:
   ```typescript
   useEffect(() => {
     const handler = (e: BeforeUnloadEvent) => {
       if (isDirty) { e.preventDefault(); e.returnValue = ''; }
     };
     window.addEventListener('beforeunload', handler);
     return () => window.removeEventListener('beforeunload', handler);
   }, [isDirty]);
   ```

5. Submit flow:
   - Show submit button only when form is valid AND in draft status
   - On submit: if in create mode, first call `createPermit()` then `submitPermit()` — but require at least 1 document (show inline error message if 0 docs on the permit)
   - After successful submit: `navigate(`/applications/${permitId}`)`

**Form fields** (in order):

a) Permit Type — `<select>` with options: Construction, Zoning Variance, Event Permit, Demolition, Renovation, Signage
b) Title — `<input type="text">` with character count hint (e.g. "43/200")
c) Description — `<textarea rows={5}` with character count hint (e.g. "247/5000")
d) Site Address — `<input type="text">` (single field — structured address is Phase 3 enhancement)
e) Contact Phone — `<input type="tel">` (optional, labeled "Contact Phone (optional)")

**Validation display**:
- On blur: `trigger(fieldName)` → show `{errors.fieldName?.message}` below each input in red text (not toast)
- On submit attempt with errors: show `<Alert variant="error">` at top with list of all failing fields
- Use the `FormField` and `Input` components from Phase 1 design system (`frontend/src/components/ui/`)

**Layout**:
- Single column, max-w-2xl centered, px-4 for mobile (no horizontal scroll at 375px)
- Section heading "New Permit Application" / "Edit Application"
- Sticky bottom action bar on mobile: Save Draft | Submit Application (two buttons)
- On desktop: buttons in the form footer
- Skeleton: 5 animated placeholder bars when loading edit mode

**Router update**: Ensure `/applications/new` and `/applications/:id/edit` routes exist (they were added in 02-05 plan but verify they're correctly set up with `<Suspense>` + lazy loading).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Permit form data (user input) sent to PATCH/POST endpoints |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-19 | Tampering | PermitFormPage form submission | mitigate | Client-side Zod validation is a UX layer; `class-validator` DTOs on the backend (`CreatePermitDto`, `UpdatePermitDto`) are the authoritative validation gate. Invalid data rejected by backend even if client validation is bypassed. |
| T-02-20 | Repudiation | Auto-save | accept | Each auto-save calls `PATCH /api/v1/permits/:id` which writes an audit log entry (`permit.updated`). All changes to draft content are traced. Residual: if auto-save fails silently (network error), the change may not be logged — accepted for v1 (user can re-save). |
| T-02-21 | Information disclosure | Draft permit in localStorage | accept | Auto-save uses the API (not localStorage) as the primary storage. FRD mentions "session expiry queues to localStorage + re-syncs" — this localStorage fallback is NOT implemented in Phase 2 (deferred complexity). Residual: if session expires, unsaved auto-save is lost. The `beforeunload` guard warns users. Accepted for v1. |
</threat_model>
