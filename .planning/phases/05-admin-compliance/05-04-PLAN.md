```typescript
export default function AdminAuditLogPage() { ... }
```

State:
- `actionFilter: string` — keyword search (empty = no filter)
- `fromDate: string` — ISO date string input value
- `toDate: string` — ISO date string input value
- `page: number` — current page (1-indexed)

On mount: call `useAdminStore().fetchAuditLog({ page: 1, limit: 20 })`.

On filter apply: call `fetchAuditLog({ action: actionFilter || undefined, fromDate: fromDate || undefined, toDate: toDate || undefined, page: 1, limit: 20 })`.

Page structure:
1. h1: "Audit Log"
2. Filter bar (horizontal flex row, wraps on mobile):
   - Action keyword: `<input type="text" placeholder="Filter by action..." aria-label="Filter by action keyword" value={actionFilter} onChange={...}>` — does NOT call API on every keystroke. Instead, "Apply Filters" button triggers the fetch.
   - From date: `<input type="date" aria-label="From date" value={fromDate} onChange={...}>`
   - To date: `<input type="date" aria-label="To date" value={toDate} onChange={...}>`
   - "Apply Filters" button: calls `fetchAuditLog` with current filter state. `aria-label="Apply audit log filters"`.
   - "Clear Filters" button: resets all filters to empty, calls `fetchAuditLog({ page: 1, limit: 20 })`.
3. Result count: `<p role="status" aria-live="polite">Showing {auditLog.length} of {meta.total} entries</p>` (shows after first load).
4. `<AuditLogTable entries={auditLog} loading={loading} />`
5. Pagination controls: Previous/Next buttons with aria-labels, disabled on boundaries. Show "Page {page} of {Math.ceil(total/limit)}" between buttons.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Admin browser sends query params (actionFilter keyword, fromDate, toDate) to GET /api/v1/admin/audit-log |
| db→render | Audit log entries (action strings, metadata JSONB, actor names) from database rendered in React table |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-15 | Elevation of privilege | `AdminAuditLogPage.tsx` — read-only page | mitigate | Audit log is read-only (GET only). Access gated by admin RBAC on backend (`@Roles('admin')` in `admin.controller.ts`, T-05-01). Page performs no mutations. |
| T-05-16 | Information disclosure | `AuditLogTable.tsx` — metadata JSONB displayed in `<pre>` | accept | The metadata field may contain internal IDs and state transitions. This is visible only to authenticated admins (role-gated). Metadata content is pre-serialized JSON from the backend (no eval or dynamic execution). Risk accepted — audit data visibility for admins is by design. |
| T-05-17 | Tampering | Action keyword filter input → SQL LIKE injection risk in backend | transfer | Frontend sends the raw string value as a query param. Backend `admin.service.ts::getAuditLog` uses TypeORM parameter binding (`WHERE action ILIKE :action`, `{ action: \`%${query.action}%\` }`) — not string interpolation. Risk transferred to backend parameterized query. |
| T-05-18 | Denial of service | fromDate/toDate range could generate large result set | mitigate | Backend `AuditLogQueryDto` enforces `limit ≤ 100` via `@Max(100)` (T-05-05). Frontend default limit is 20. No date range generates more than 100 rows per page. |
</threat_model>
