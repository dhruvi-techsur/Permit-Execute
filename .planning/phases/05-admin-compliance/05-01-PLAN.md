**Methods:**

`getAllApplications(query: { status?, page?, limit?, reviewerId?, applicantId? })`:
- Use TypeORM QueryBuilder on `permit_applications`
- LEFT JOIN `users` AS `applicant` ON `applicant.id = app.applicant_id`
- LEFT JOIN `users` AS `reviewer` ON `reviewer.id = app.reviewer_id`
- Apply WHERE filters for status, reviewerId, applicantId when provided
- ORDER BY `app.created_at DESC`
- Return: `{ data: PermitApplicationSummary[], meta: { total, page, limit } }`
- PermitApplicationSummary shape: `{ id, title, permitType, status, submissionDate, applicantName: applicant.full_name, reviewerName: reviewer?.full_name ?? null, createdAt }`

`getAllUsers(query: UserListQueryDto)`:
- QueryBuilder on `users` table
- Apply WHERE filters for role, isActive when provided
- ORDER BY `created_at DESC`
- Paginate. Return `{ data: AdminUserDto[], meta }` — AdminUserDto omits `password_hash`
- AdminUserDto: `{ id, email, fullName, role, isActive, createdAt, updatedAt }`

`createUser(dto: CreateUserDto)`:
- Hash password with bcrypt (saltRounds: 10, use bcrypt from `bcryptjs`)
- Insert new User row with is_active=true
- Return AdminUserDto (without password_hash)
- Throw `ConflictException` if email already exists (catch unique violation code `'23505'`)

`updateUser(id: string, dto: UpdateUserDto)`:
- Find user by ID, throw `NotFoundException` if not found
- Merge DTO fields (skip undefined). If `fullName` provided, map to `full_name` column. If `role` provided, update role. If `isActive` provided, update `is_active`.
- Save and return AdminUserDto
- Do NOT allow updating `email` or `password` via this endpoint

`deactivateUser(id: string)`:
- Find user by ID, throw `NotFoundException` if not found
- Set `is_active = false`, save
- Return `{ message: 'User deactivated' }`

`assignReviewer(applicationId: string, dto: AssignReviewerDto)`:
- Verify application exists, throw `NotFoundException` if not
- Verify reviewer user exists and has role='reviewer', throw `BadRequestException` if role mismatch
- Set `reviewer_id = dto.reviewerId`, save
- Return `{ applicationId, reviewerId: dto.reviewerId, reviewerName: reviewer.full_name }`

`getAuditLog(query: AuditLogQueryDto)`:
- QueryBuilder on `audit_logs` LEFT JOIN `users` AS actor ON `actor.id = audit_logs.actor_id`
- LEFT JOIN `permit_applications` AS app ON `app.id = audit_logs.application_id`
- Apply WHERE filters: applicationId, actorId, action (ILIKE `%action%`), fromDate (created_at >= fromDate), toDate (created_at <= toDate)
- ORDER BY `audit_logs.created_at DESC`
- Return `{ data: AuditLogEntry[], meta }` — AuditLogEntry: `{ id, actorId, actorName: actor.full_name, actorEmail: actor.email, applicationId, action, metadata, createdAt }`

All pagination: `skip = (page-1)*limit`, `take = limit`, return `meta: { total, page, limit }`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Untrusted HTTP request bodies, query params, and path params from admin browser sessions crossing into admin controller handlers |
| db→render | User records (including sensitive fields) from the database crossing into API response serialization |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Elevation of privilege | `admin.controller.ts` — all endpoints | mitigate | `@UseGuards(JwtAuthGuard, RolesGuard)` + `@Roles('admin')` on the controller class (not per-method) ensures every admin endpoint requires a valid admin JWT; guard logic lives in `backend/src/auth/guards/roles.guard.ts` |
| T-05-02 | Information disclosure | `admin.service.ts::getAllUsers` + `createUser` + `updateUser` | mitigate | `password_hash` column is never selected in QueryBuilder SELECT or returned in AdminUserDto; use explicit column selection (`SELECT u.id, u.email, u.full_name, u.role, u.is_active, u.created_at, u.updated_at`) — never `SELECT *` on the users table |
| T-05-03 | Tampering | `admin.controller.ts::deactivateUser` — `DELETE /api/v1/admin/users/:id` | mitigate | `ParseUUIDPipe` on `:id` param in `admin.controller.ts::deactivateUser` prevents non-UUID injection; service performs `findOne` before soft-delete, ensuring operation is scoped to a valid user row |
| T-05-04 | Tampering | `admin.service.ts::assignReviewer` | mitigate | Service verifies target user has `role='reviewer'` before assignment, preventing assignment of non-reviewer users; `ParseUUIDPipe` on both `applicationId` and `reviewerId` prevents UUID injection |
| T-05-05 | Denial of service | `GET /api/v1/admin/applications`, `GET /api/v1/admin/users`, `GET /api/v1/admin/audit-log` | mitigate | `limit` query param capped at 100 via `@Max(100)` in all query DTOs; NestJS `ValidationPipe` (global) rejects invalid limit values with 400 |
| T-05-06 | Repudiation | All mutating admin endpoints (POST/PATCH/DELETE users, PATCH assign-reviewer) | accept | Audit logging for admin actions is not in scope for Phase 5 (AuditService.createEntry calls are on the permit workflow path, not admin user management); admin actions are logged at the database level via updated_at timestamps. Accepted risk owned by project; can be added in v2 |
</threat_model>
