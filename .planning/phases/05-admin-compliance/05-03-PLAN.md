- Empty state: "No users found" centered text.
- Accessible: `<th scope="col">` for all headers, `<table>` has `aria-label`.

**`frontend/src/pages/admin/AdminUsersPage.tsx`**:

```typescript
export default function AdminUsersPage() { ... }
```

State:
- `selectedUser: AdminUserDto | null` — for edit modal
- `showCreateModal: boolean` — for create modal
- `roleFilter: string` — 'all' | 'applicant' | 'reviewer' | 'admin'
- `activeFilter: string` — 'all' | 'active' | 'inactive'
- `page: number` — current page (1-indexed)

On mount + on filter/page change: call `useAdminStore().fetchUsers({ role: roleFilter !== 'all' ? roleFilter : undefined, isActive: activeFilter !== 'all' ? activeFilter === 'active' : undefined, page, limit: 20 })`.

Page structure:
1. Header row: h1 "User Management" + "Create User" button (primary Tailwind style, `aria-label="Create new user"`)
2. Filter bar:
   - `<select aria-label="Filter by role">` with options: All Roles, Applicant, Reviewer, Admin
   - `<select aria-label="Filter by status">` with options: All Status, Active, Inactive
3. `<UsersTable users={users} loading={loading} onEdit={setSelectedUser} onDeactivate={handleDeactivate} />`
4. Pagination controls (Previous/Next buttons with aria-labels, disabled on boundaries)
5. `<CreateUserModal open={showCreateModal} onClose={() => setShowCreateModal(false)} onCreated={handleCreated} />`
6. `<EditUserModal user={selectedUser} onClose={() => setSelectedUser(null)} onUpdated={handleUpdated} />`

**Deactivate flow** (`handleDeactivate`):
- `window.confirm('Deactivate {user.fullName}? They will no longer be able to log in.')` 
- If confirmed: call `adminApi.deactivateUser(user.id)`, then update local store state by replacing the user with `{ ...user, isActive: false }` (use `useAdminStore().fetchUsers(...)` to re-fetch, or update store state directly).
- On error: show toast/alert.

NOTE: `window.confirm` is acceptable here as a simple confirmation for an admin action — no custom modal needed for deactivation.

**`handleCreated`**: After create modal success, re-fetch users: `fetchUsers({ page: 1 })`, reset page to 1.
**`handleUpdated`**: After edit modal success, re-fetch users with current filters.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Admin form inputs (email, password, role, fullName) crossing from browser form into POST /api/v1/admin/users and PATCH /api/v1/admin/users/:id |
| db→render | User records from API responses rendered in React table — user-supplied fullName and email displayed as text content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-11 | Elevation of privilege | `AdminUsersPage.tsx` — create user with role='admin' | mitigate | Backend `CreateUserDto.role` validates via `@IsEnum(['applicant','reviewer','admin'])` in `admin.service.ts::createUser`; only a currently-authenticated admin can call this endpoint (T-05-01). Frontend role dropdown offers all three options — this is by design for admin capability. |
| T-05-12 | Information disclosure | `CreateUserModal.tsx` — password field | mitigate | Password field is `type="password"` by default; show/hide toggle uses a button that sets `type="text"` locally. Password value is sent over HTTPS to backend and immediately bcrypt-hashed in `admin.service.ts::createUser`; never returned in any API response. |
| T-05-13 | Tampering | `EditUserModal.tsx` — only changed fields sent | accept | Sending only changed fields reduces payload but does not create a security risk — backend `UpdateUserDto` validates all fields independently. Backend is the authority on which fields are permitted. Risk accepted; no additional guard needed on the frontend partial-update logic. |
| T-05-14 | Information disclosure | `UsersTable.tsx` — renders user email + fullName | transfer | Data is already scoped to admin-only API endpoint (T-05-01). React renders as text content (not `dangerouslySetInnerHTML`), preventing XSS. No additional guard needed; risk transferred to backend RBAC. |
</threat_model>
